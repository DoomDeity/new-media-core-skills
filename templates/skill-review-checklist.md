# Skill 审查检查清单 / Skill Review Checklist

> 与 `docs/skill-quality-playbook.md` 配套。创建前完成"创建前检查"，生成后完成"创建后检查"。
> 任一项不通过不得进入下一阶段。

## 创建前检查

1. 唯一职责是否明确（一句话说清"做什么"和"绝不做什么"）？
2. 相邻 Skill 是谁（读 `docs/skill-inventory.md` 与相邻源文件）？
3. 哪些请求绝对不应触发（写入 When NOT to Use 并引用相邻 Skill）？
4. 输入是否真实可获得（是否需要联网、文件或用户样本）？
5. 适合该任务的证据来源是什么（行业研究 / 竞品分析 / 评论分析各有不同）？
6. 有哪些隐私与合规风险（用户材料是否需匿名化、是否可提交外部工具）？
7. 是否需要 `references` / `scripts` / `assets`（不为形式完整而建空目录）？

## 创建后检查

1. `description` 是否包含"做什么 + 何时用 + 关键禁止边界"？
2. When to Use 与 When NOT to Use 是否冲突？
3. Workflow 是否可以真实执行（命令式、无悬空步骤）？
4. Output 是否直接可用（结构完整、可落盘）？
5. 是否存在无来源数字（如凭空的市场量级）？
6. 是否存在推断冒充事实（是否标注"推断/未知"）？
7. 是否存在伪精确（小样本是否用了精确百分比而非实际数量）？
8. 示例是否虚构、脱敏且数据自洽（合计可核对）？
9. 是否写死具体工具名（应保留可迁移性）？
10. 是否越过 Skill 边界（不该做的交相邻 Skill）？
11. 是否与相邻 Skill 重复（职责是否清晰分离）？
12. 是否考虑缺失输入（假设 vs 追问 vs 不编造）？
13. 是否完成实际测试（十类测试，记录于 `tests/<编号>-<name>/test-record.md`）？
14. 是否仍处于开发阶段（未确认定稿前不生成 README / 发布包 / dist）？
15. 是否错误地提前创建发布文件（定稿后才补齐辅助文件）？
