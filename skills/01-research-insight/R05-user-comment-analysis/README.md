# 用户评论分析（user-comment-analysis）

| 项目 | 内容 |
|------|------|
| 中文名称 | 用户评论分析 |
| 英文名称 | user-comment-analysis |
| Skill 编号 | R05 |
| 当前版本 | 1.3.0 |
| 一句话介绍 | 从评论 / 弹幕 / 私信文本提取痛点信号、需求推断、情绪与意图，默认只分析你给的材料。 |

## 解决的问题

有一批评论 / 弹幕 / 私信，想提炼痛点信号、需求推断、情绪和高频问题；做内容复盘、产品优化、选题灵感挖掘时缺结构化归纳。

## 适合人群

新媒体运营、内容创作者、产品经理、用户运营、需要做评论复盘的人。

## 适用场景

- 有一批评论 / 弹幕 / 私信 / 反馈，想提炼痛点信号、需求推断、情绪、高频问题。
- 做内容复盘、产品优化、选题灵感挖掘。

## 不适用场景

- 主动通过问卷 / 访谈挖掘需求 → 用 **用户需求挖掘（R06）**。
- 识别可能升级的负面舆情与争议 → 用 **舆情风险识别（R08）**。
- 撰写对单条评论的公开回复 → 用 **评论区回复（P03）**。
- 分类并制定负面评论处理方案 → 用 **负面评论处理（P04）**。

## 核心能力

隐私匿名化；清洗去噪（保留重复评论为频次信号）；情绪分层（正 / 中 / 负）；评论意图分类（与情绪两套独立分类）；痛点信号与需求推断；高频问题汇总；行动建议。

## 必要输入

1. 评论文本（原始粘贴或整理后的列表）。
2. 可选上下文：对应内容主题、发布平台、分析目的、样本周期。

无上下文仅做通用归纳并标"假设"；评论文本完全为空则集中询问一次。

## 主要输出

Markdown 分析，至少包含：样本概况（数量 / 周期 / 平台 / 匿名化说明）、情绪分布（正 / 中 / 负，标样本量）、评论意图分布（与情绪独立）、痛点信号与需求推断清单（排序，标推断）、高频问题 TOP（带次数或频次）、行动建议（1–3 条）。

## 简短使用示例

用户提供 20 条视频评论，多问"资料在哪领""求模板"。

输出：匿名化（无个人信息）→ 情绪（正向 6 / 中性 12 / 负向 2，合计 20，小样本用实际数量）→ 意图（求资料 8 / 提问 4 / 认可 3 / 购买咨询 2 / 其他 3，合计 20，与情绪分开单列）→ 痛点信号（找不到模板入口）/ 需求推断（想要可复用模板）→ 高频（"资料链接""怎么领"）→ 建议（置顶资料领取 + 做模板合集视频）。

## 与相邻 Skill 的区别

R05 只归纳**已提供的评论文本**；R06 主动挖掘需求（问卷 / 访谈）；R08 识别舆情风险；P03 写单条公开回复；P04 处理负面评论方案。输入输出不同。

## 事实、推断与未知的处理规则

三级标注：已观察事实（用户原话）/ 证据支持的推断（标"推断"）/ 未知（标"待验证"）。来源优先级：用户提供的原始评论、弹幕、私信和反馈样本 > 用户补充的业务上下文 > 用户明确要求后检索的公开背景（默认不主动外部搜索）。小样本不输出伪精确比例，改用"少量样本 / 频次信号"。

## 数据、来源或样本限制

情绪与意图属"推断"，须标注基于样本，不夸大代表性。痛点信号 / 需求推断非确证事实，须标"推断"。小样本优先用实际数量（如"正 6 / 中 4 / 负 2"），不输出无必要精确百分比。引用代表评论须来自用户实际输入并继续匿名化，不得编造。

## 隐私与安全边界（重点）

- 含个人姓名、联系方式、账号 ID 等可识别信息的评论，**先匿名化**（替换为"用户 A"等）再分析。
- 未经匿名化，不得将含真实姓名、手机号、邮箱、账号 ID、订单号、地址或私信可识别信息的原始材料提交外部搜索或第三方工具。
- 默认只分析用户提供的材料；仅当用户明确要求补充公开背景时，才调用外部工具检索，且涉及个人信息的材料仍须先匿名化。
