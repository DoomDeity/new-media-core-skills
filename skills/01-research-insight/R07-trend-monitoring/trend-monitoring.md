# 热点趋势监测

## When to Use

- 用户想了解近期行业、平台、内容或社会层面的热点与变化信号。
- 用户要判断某个话题是否值得跟进、处在什么阶段、和自己业务有什么关系。
- 用户说："最近 XX 是不是个趋势""这个热点要不要跟""帮我看看行业最近在聊什么""XX 平台最近有什么变化信号"。

## When NOT to Use

- 系统研究行业长期结构、产业链、市场机会 → 用 `industry-research`（R01）。
- 监测平台规则、政策条款、功能机制变化 → 用 `platform-rule-monitoring`（P05）。
- 识别负面舆情、争议议题与升级风险 → 用 `public-opinion-risk-identification`（R08）。
- 只归纳一批评论的痛点与情绪 → 用 `user-comment-analysis`（R05）。
- 直接生成选题或内容 → 用 `topic-generation`（C04）等。

## Inputs

必要信息：

1. 关注领域 / 平台 / 业务背景（如"小红书职场内容""抖音本地生活"）。
2. 具体想核验的话题或信号（可选；不提供则做广谱扫描）。
3. 时间窗口（默认近 30–90 天）。
4. 用户业务与目标（用于判断相关性）。

缺失处理：业务背景缺失且可合理假设时，标注"假设：默认面向新媒体运营者"后继续；若完全无关注领域，集中询问一次（仅问关注哪个行业 / 平台）。

## 缺失信息处理

- 有话题但缺业务背景：仍给趋势判断，相关性标注"假设"。
- 话题信息过期风险：涉及"当前热点"必须调用当前已配置的搜索、浏览器、平台数据或文件分析工具检索最新资料；不依赖训练记忆，检索结果标注来源与日期。
- 单点信号（一条爆款 / 一篇报道 / 少量讨论）：不升级为趋势，归类"噪声或证据不足"，说明证据薄弱。
- 会实质影响相关性判断（如完全无业务背景且无法确定）时：集中询问一次。

## Workflow

1. 明确监测口径：确定领域、平台、时间窗与用户业务目标。
2. 检索与核验最新信号：若用户提供近期一手材料（如粘贴的近期公开讨论、平台数据截图、媒体链接），以材料为准并标注其日期，必要时调用当前已配置的搜索、浏览器、平台数据或文件分析工具补充核验；若用户仅给话题、无近期材料，必须主动检索最新公开资料。所有信号记录来源与检索日期。
3. 信号归集：把零散信号按主题归类，区分"单次事件"与"重复出现的同类信号"。
4. 逐条评估（每条趋势至少包含以下字段）：
   - 观察到的信号（具体现象，标「事实」）
   - 来源与日期（媒体 / 平台 / 公开讨论，含检索日期）
   - 热度阶段（起步 / 升温 / 高峰 / 回落 / 稳定）
   - 与用户业务的相关性（高 / 中 / 低 + 依据）
   - 持续性判断（短期 / 中期 / 待验证）
   - 状态标注（事实 / 推断 / 未知）
5. 分类输出：短期热点 / 正在形成的趋势 / 已趋于稳定的变化 / 噪声或证据不足。
6. 相关性排序：按"相关度 × 持续性 × 可行动性"定性排序，附依据。
7. 给用户的建议：给出"是否值得跟进、如何验证"的建议，但**不替用户决定是否追热点**。
8. 自检：是否有无来源数字？旧信息是否冒充当前趋势？单点信号是否被夸大？推断是否标清？

## Output

Markdown 监测简报，至少包含：

- 监测口径（领域 / 平台 / 时间窗 / 业务背景，含检索日期）
- 趋势 / 热点清单（每条含上述 6 个字段）
- 分类视图（短期热点 / 正在形成的趋势 / 已趋稳定的变化 / 噪声或证据不足）
- 相关性排序（定性 + 依据）
- 给用户的建议（标「建议」；验证方式、跟进姿势，不替用户拍板）
- 明确"结论基于检索到的公开信号，热度阶段与持续性含推断，需持续观察"

如需落盘，文件名如 `热点趋势监测-领域-YYYYMMDD.md`。

## Evidence / Quality Rules

- 实时优先：当前热点必须主动检索最新公开资料，不默认依赖训练记忆；引用须标来源与日期（如"来源：XX 媒体，检索于 2026-07-20"）。
- 来源优先级：一手平台数据 / 官方公告 > 权威媒体 / 行业报告 > 聚合内容 / 社媒讨论 > 单条爆款内容。低优先级须降级标注。
- 三级标注：区分「事实（已观察信号）」「推断（趋势与持续性判断）」「未知（待验证）」。
- 禁止虚构：不编造搜索量、热度指数、增长率、参与人数等无来源数字；无可靠口径明确"无法量化热度"。
- 证据门槛：不因为单条爆款内容、单个媒体报道或少量讨论就断言形成趋势；跨平台 / 多来源重复出现才提升为"正在形成的趋势"。
- 样本说明：信号数量、覆盖平台、周期须标注；小样本不输出伪精确比例。

## Boundaries

- 关注近期变化与趋势信号，不承担完整行业研究（那是 R01）。
- 平台规则、政策、功能机制变化归 P05，不在此处理。
- 负面舆情与升级风险归 R08，本 Skill 只做中性趋势判断。
- 热度阶段、持续性、相关性均为「推断」，须说明依据，不夸大确定性。
- 不替用户决定是否追热点，只给判断依据与建议。
- 若材料涉及用户私信 / 个人信息，先匿名化，不提交未处理隐私材料到外部工具。

## Examples

例 1（虚构示例，仅示范结构，不代表真实事实）：用户运营小红书职场账号，问"AI 求职是不是趋势"。检索到：3 个平台近 30 天出现"用 AI 改简历"讨论、2 篇行业媒体报道、1 个招聘平台上线相关功能。输出：信号（"多平台出现 AI 简历工具讨论"，标事实，来源：公开讨论 + 媒体报道，检索 2026-07-20）→ 热度阶段（升温）→ 相关性（高，依据：用户受众为求职者）→ 持续性（中期，标推断）→ 状态（推断）→ 分类（正在形成的趋势）。另有一条单账号爆款视频"AI 面试通关"点击高但无后续讨论，归类"噪声或证据不足"，说明仅单点信号。建议（标建议：可先做 1–2 篇测评内容验证受众反应，不盲目all-in）。

例 2（虚构示例，旧信息冒充当前）：用户提供一篇 2024 年的"XX 玩法爆发"文章询问是否现在该跟。输出：先检索当前信号，发现近 90 天无新增讨论，判定为"已趋于稳定的变化 / 退潮"，明确告知"该信号源于 2024，当前无延续证据，不视为当前趋势"，避免把旧信息当热点。

## 最终自检

- 触发准确：仅覆盖近期热点与趋势信号，已互指 R01 / P05 / R08 / R05 / C04。
- 边界清晰：不做行业长期研究、不处理规则变化、不识别舆情风险、不代写内容。
- 无来源数字：未编造搜索量 / 热度指数；示例均为虚构且自洽。
- 旧信息未冒充当前：例 2 明确区分历史信号与当前趋势。
- 单点信号未夸大：例 1 将单条爆款归为噪声。
- 推断已标注：热度阶段 / 持续性 / 相关性均标「推断」并附依据。
- 未写死工具：仅用"当前已配置的搜索、浏览器、平台数据或文件分析工具"。
- 未越界：只给建议，不替用户决定是否追热点。
