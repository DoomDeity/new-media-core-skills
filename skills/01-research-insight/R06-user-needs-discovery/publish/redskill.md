- **一句话介绍（30 字内）**：从多源材料区分真需求与方案偏好，不虚构结论。

## 主要解决的问题

手握访谈、问卷、客服、评论等多类材料却理不清用户真正要什么；容易把用户说的"加个按钮"误当成底层需求；没有真实材料时被要求给出需求结论。

## 3—5 个典型使用场景

1. 归纳多源用户研究材料，提炼显性需求与潜在需求。
2. 做需求优先级排序、需求地图。
3. 规划下一步用户研究（访谈提纲、问卷框架）。
4. 缺少真实材料时，先产出研究设计而非拍脑袋结论。

## 适合人群

产品经理、用户研究员、新媒体 / 内容运营、增长与体验团队、需要做需求优先级判断的人。

## 推荐调用语句

"帮我把这些访谈和问卷里的需求理一理，按优先级排一下。"
"用户到底想要什么，帮我拆一下底层需求。"
"我没材料，先给我一套需求访谈提纲和问卷框架。"

## 输入要求

一手用户材料（访谈 / 问卷 / 客服 / 私信 / 评论 / 搜索词 / 工单）、业务背景、分析目的。完全无材料时只做研究设计，不虚构需求结论。

## 输出内容

需求分层表（原始表达 → 方案偏好 → 场景 → 底层需求[推断] → 待验证假设）、显性 / 潜在需求清单、需求强度与业务价值分离表、样本边界、下一步建议，并明确"基于所提供材料、不代表全量用户"。

## 不适用场景

只归纳评论痛点情绪（归 R05）、完整行为路径分析（归 D04）、直接生成选题内容（归 C 类）、账号定位（归 C01）、销售话术（归 G 类）。

## 事实和数据边界

用户提出的解决方案偏好不自动等同底层需求；需求强度（真实、影响多大）与业务价值（是否符合目标）分开，业务优先级判断属推断并说明依据；多源不相加比例，同用户跨源不机械计多用户；小样本不输出伪精确百分比。

## 隐私及合规提醒

含姓名、联系方式、账号 ID、订单号等先匿名化再分析；未匿名化不提交外部工具；未经授权不调用付费数据接口。

## 发布文案（适合小红书 / RedSkill 介绍页）

做产品或内容最怕把用户一句"加个按钮"当成真需求，最后做了个没人用的功能。这个用户需求挖掘 Skill 帮你把模糊的"用户想要什么"拆成四层：用户原话、他提的方案偏好、真实使用场景、以及背后真正的底层需求——每一层都标清楚这是事实、推断还是还没验证。它没有真实材料时不会替你编结论，而是先给你一套研究方法；有材料时则把"需求是否真实"和"是否符合当前业务目标"分开算，不把优先级高错当成需求已经证实。适合产品经理、运营和研究者。
