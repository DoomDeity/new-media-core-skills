# 用户需求挖掘（user-needs-discovery）

| 项目 | 内容 |
|------|------|
| 中文名称 | 用户需求挖掘 |
| 英文名称 | user-needs-discovery |
| Skill 编号 | R06 |
| 当前版本 | 1.0.0 |
| 一句话介绍 | 从多类用户研究材料中区分原始表达、表层诉求与底层需求，识别优先级，缺失真实材料时只做研究设计、不虚构结论。 |

## 解决的问题

手握访谈、问卷、客服记录、搜索词、评论、行为线索等多类材料，却理不清用户真正要什么；容易把用户随口说的"加个按钮"误当成底层需求；分不清"需求是否真实"与"是否符合当前业务目标"；没有任何真实材料时，被要求给出用户结论。

## 适合人群

产品经理、用户研究员、新媒体 / 内容运营、增长与体验团队、需要做需求优先级判断的人。

## 适用场景

- 归纳多源用户研究材料，提炼显性需求与潜在需求。
- 做需求优先级排序、需求地图或下一步研究规划。
- 规划下一步用户研究（访谈提纲、问卷框架、搜索词清单）。

## 不适用场景

- 只拿到一批评论 / 弹幕 / 私信，想归纳痛点、情绪、高频问题 → 用 `user-comment-analysis`（R05）。
- 要做完整用户行为路径、漏斗与数据异常分析 → 用 `user-behavior-analysis`（D04）。
- 要直接生成选题、内容栏目或文案 → 用 `topic-generation`（C04）/ `content-pillar-planning`（C02）等。
- 要做账号定位或目标人群定义 → 用 `account-positioning-analysis`（C01）。
- 要设计销售话术或转化路径 → 用 `sales-talk-track`（G04）/ `content-acquisition-path`（G01）。

## 核心能力

多源去重（同用户跨源不机械计多用户）；逐层拆解用户表达（原始表达 / 表层诉求 / 使用场景 / 底层需求 / 待验证假设）；区分显性需求与解决方案偏好（用户提出的"按钮 / 功能"只是表层诉求，不直接等同需求）；拆开需求强度与业务价值（证据强度 / 影响范围 / 紧迫性 / 业务相关性 / 验证状态分别输出）；样本边界说明；无材料时只做研究设计、不虚构用户结论。

## 必要输入

1. 一手用户材料：访谈记录 / 转录、问卷原始回答、客服对话、私信、评论、搜索词列表、工单。
2. 业务背景：产品 / 内容 / 服务当前形态、目标人群、所处阶段（验证期 / 增长期）。
3. 已有研究结论或假设（可选）。
4. 分析目的：排序需求 / 建需求地图 / 规划下一步研究。

缺失且可合理假设时，标注"假设：默认分析目的是需求优先级梳理"后继续；若完全没有用户材料，本 Skill 只帮助设计研究方法、访谈提纲、问卷框架或搜索词清单，不得虚构任何用户结论。

## 主要输出

Markdown 报告，至少包含：样本概况（各来源数量 / 周期 / 平台 / 招募方式 / 匿名化 / 去重说明）、需求分层表（原始表达 → 表层诉求 / 方案偏好 → 使用场景与目标 → 底层需求[推断，含回溯] → 待验证假设）、显性需求清单、潜在需求清单、使用场景（主 / 边缘）、阻碍因素、需求强度与业务价值分离表、样本边界、未验证假设、下一步建议，并明确"本结论基于所提供的 X 份材料，不代表全量用户"。

## 简短使用示例

用户说："帮我把这些访谈和问卷里的需求理一理，按优先级排一下。"

输出：需求分层表（原始表达 → 方案偏好 → 场景 → 底层需求[推断] → 待验证假设）；样本内证据状态分别标注（如"权益汇总"=样本内部分支持，依据访谈 3/8 独立受访者提及，客服相似表达可能重复仅作补充信号、不提高独立样本数量）；业务相关性=高，但业务执行优先级建议=高须明确为综合建议、不等于需求已总体证实；"批量导出"因仅单点证据标待验证；建议优先验证"权益汇总"需求（补充访谈 / 低保真原型 / 小范围可用性验证）。

## 与相邻 Skill 的区别

R05 只归纳评论的痛点、情绪、高频问题，不做需求分层；D04 看完整行为路径与数据异常，不负责需求归纳；C 类生成选题 / 内容方案。R06 只归纳用户已提供的材料、识别需求与优先级，不替代行为分析，不代写选题 / 产品方案 / 话术。

## 事实、推断和未知的处理规则

状态标签统一为四类：

- **事实**：用户原话、明确表达的目标 / 问题 / 痛点（含用户提出的解决方案偏好，但标注为"表层诉求 / 方案偏好"，不直接当需求事实）。
- **推断**：底层需求、使用场景、阻碍因素（须能回溯到具体原始材料与推断过程）。
- **假设 / 未知 / 待验证**：环境或外部未证实的事项。
- **建议**：给用户的行动。

不使用"三级 + 一类"等易误解分类数量名称。

## 数据、来源或样本限制

来源优先级：用户一手材料（访谈 / 问卷 / 客服 / 私信 / 评论原样本）> 用户补充的业务上下文 > 明确要求的公开检索背景（默认不主动外联）。多源去重：同一用户跨源重复表达不机械计为多用户；同一事件多次转述不机械累计为多个独立信号；无法识别重复时标"可能存在重复样本"；不同来源数量、频次、样本偏差分别呈现，不直接相加计算总体比例。样本说明：给出各来源数量、周期、平台、招募方式；小样本不输出伪精确百分比，改用实际数量与"样本有限"限定。

## 隐私与安全边界

含姓名、联系方式、账号 ID、订单号等可识别信息的材料，先做匿名化（替换为"用户 A"等）再分析；未匿名化材料不提交外部检索 / 第三方工具。未经用户授权不调用付费数据接口；用户已授权并配置的工具（搜索 / 浏览器 / 平台数据 / 文件分析）可以使用。分析材料仅用于本次研究，不做无关留存。
