# 质量复盘手册 / Skill Quality Playbook

> 从 R01–R05（行业研究、竞品定位分析、竞品内容分析、竞品营销策略拆解、用户评论分析）
> 从初稿到 1.3.0 定稿的多轮返工中沉淀的永久规范。
> 目标：后续每个 Skill 第一次生成就尽量接近当前定稿质量，不再重复已确认的错误。
> 本文件与 `templates/skill-review-checklist.md` 配套使用；详细强约束见 `docs/design-spec.md` 与 `docs/test-spec.md`。

---

## 1. 我们为什么反复返工（主要原因）

1. **初稿只具备结构，没有足够严格的证据规则。** 先有骨架，再补"事实/推断/未知"标注，导致早期版本把推断当事实。
2. **不同任务错误复用同一套来源优先级。** 行业研究的来源体系被直接套到竞品分析，忽略了两者证据性质不同。
3. **研究型 Skill 容易把推断写成事实。** 尤其是"行业趋势""玩家格局"这类结论性内容。
4. **容易生成无来源数字和伪精确比例。** 如直接写"市场规模 1200 亿"而无口径与方法。
5. **小样本仍输出精确百分比。** 样本量 12 时给"25% 负向"，精确度和样本不匹配。
6. **竞品分析容易推测内部投放、转化和成交信息。** 把公开可见的入口当成"已证实的转化路径"。
7. **示例使用真实品牌或未经核实事实。** 一旦公开就会变成"引用了未核实信息"。
8. **示例中的数量和分类不自洽。** 如"情绪 正6/中12/负2"合计却是 12，暴露统计错误。
9. **情绪、意图、主题等不同维度被混在一起。** 一个标签同时承载多种含义，输出不可信。
10. **广告、活动和异常内容分类不清。** "商业内容""非自然内容""异常内容"边界模糊。
11. **写死 RedFox、AnySearch 等具体工具名。** 导致 Skill 不可移植，换了环境就失效。
12. **用户材料与公开检索之间缺少隐私边界。** 没说清"哪些材料可以提交外部工具"。
13. **相邻 Skill 职责边界不够清楚。** 竞品三类分析（定位/内容/营销）互相重叠，触发易串。
14. **开发源文件与平台 `SKILL.md` 入口混为一谈。** 出现两份正文、修改分叉。
15. **编号、目录、文件名和 frontmatter 规则反复变化。** 早期在源文件名里带编号、后又移除，造成历史包袱。
16. **尚未确认定稿时过早生成 README 和发布包。** 核心内容还在变，辅助文件反复返工。
17. **references、scripts、assets 容易机械创建。** 为"形式完整"建空目录，反而增加维护成本。
18. **版本状态和"定稿状态"混为一谈。** 把"完成测试"等同于"已发布/已定稿"，对外口径失真。

## 2. 后续 Skill 第一次生成就必须遵守的原则

1. **先读取 Skill 清单和相邻 Skill。** 不孤立设计一个 Skill；清单见 `docs/skill-inventory.md`。
2. **先确认唯一职责和禁止边界。** 一句话说清"做什么"和"绝不做什么"，并在 When NOT to Use 引用相邻 Skill。
3. **先选择适合该任务的证据来源体系。** 行业研究、竞品分析、评论分析各有不同来源优先级（见第 3 节）。
4. **明确事实、推断、未知、建议。** 用统一前缀标注，不把推断或假设写成事实。
5. **不编造数据。** 无可靠来源与估算方法时，不输出市场量级或精确占比。
6. **不制造伪精确。** 小样本优先报告实际数量、频次和"样本有限"的限定，而不是精确百分比。
7. **小样本优先使用实际数量和频次信号。** 如"负面评论 3 条 / 共 20 条样本，结论受样本量限制"。
8. **示例必须虚构、脱敏且数据自洽。** 所有数字内部可加总、可核对；不使用真实品牌或真实个人。
9. **不写死非必要工具名。** 需要外部能力时用"公开检索/平台数据接口"等通用表述，保留可迁移性。
10. **信息不足时只集中询问一次。** 只问会实质影响结果的关键项，不逐条追问，也不静默编造。
11. **默认保持可迁移性。** 同一逻辑在不同 Agent / 环境下都应可用，不依赖特定本地路径或账号。
12. **不把观察到入口等同于证明真实贡献。** 公开可见的落地页/活动入口 ≠ 已证实的转化路径。
13. **不把公开产品阶梯等同于真实成交路径。** 价格档位、会员等级不是成交证据。
14. **涉及用户材料时先定义隐私和外部工具边界。** 原始含个人信息材料先匿名化，不提交外部搜索或第三方工具。
15. **Workflow 最后执行自检。** 检查是否有无来源数字、推断冒充事实、维度混淆、示例不自洽。
16. **不扩大 Skill 职责。** 边界外的事交给相邻 Skill，不在本 Skill 里顺手做。
17. **辅助文件只在项目所有者确认定稿后生成。** 定稿前只维护开发源文件与测试记录。
18. **运行包必须由源文件生成。** 修改只改源文件，再跑 `scripts/build_dist.py`，不得直接改 `dist/SKILL.md`。

## 3. 任务类型差异（不能机械共用同一套规则）

不同 Skill 的证据性质不同，规则不能一刀切：

- **行业研究**：重视官方统计、行业报告、上市公司披露；必须标注统计口径与时间有效性；无可靠口径时不输出市场量级。
- **竞品定位分析**：重视竞品官方表达（官网、公开介绍、招股书）；结论与证据一一对应；不推测其内部组织。
- **竞品内容分析**：重视样本周期、表现口径（曝光/互动定义）、内容分类；区分常规内容、商业与运营内容、疑似异常内容。
- **竞品营销策略拆解**：重视逐节点证据状态——已观察 / 证据支持的推断 / 未知，不得只用前两类而省略"未知"。
- **用户评论分析**：重视原始样本与匿名化；严格区分情绪、意图、主题等维度；隐私边界最高优先。
- **后续内容生成 / 增长 / 数据 / 管理类 Skill**：也必须建立适合自身任务的证据与边界规则，不能套用研究型模板。

通用原则（第 2 节）跨类型适用；具体来源体系与分类口径（第 3 节）按类型定制。模板只约束结构与质量，不替代具体业务设计。

---

## 4. 如何使用本手册

- 创建新 Skill 前：读 `docs/skill-inventory.md`、`docs/skill-quality-playbook.md`、`templates/skill-review-checklist.md` 与相邻 Skill 文件，完成"创建前检查"。
- 生成初稿后：用检查清单完成"创建后检查"，并用 `templates/skill-source.template.md` 对照结构与证据规则。
- 实际测试通过后，再提交项目所有者审查；确认定稿后才生成 README / 发布包 / dist。
- R01–R05 已作为通过验证的参考实现，但其业务逻辑不得复制到不相关 Skill。
