# Skill 测试与验收规范

> 状态：已落地。每个 Skill 创建后必须按本规范完成十类测试与状态更新，未通过不得标记为完成。
> 最近更新：2026-07-20

## 1. 测试目标

确保每个 Skill：可独立调用、触发准确、边界清晰、输出可用、不编造事实。

## 2. 测试类型与通过标准

每个 Skill 原则上必须通过以下全部十类测试：

| 类型 | 目的 | 通过标准 |
|------|------|---------|
| 1. 触发测试 | 典型用户说法能正确触发该 Skill | 给定 3+ 条典型语料，Skill 被正确命中；description 命中逻辑清晰 |
| 2. 防误触发测试 | 相近但不属于该 Skill 的请求不触发 | 给定 3+ 条易混淆语料，Skill 不被误触发；必要时由更合适的相邻 Skill 接管 |
| 3. 相邻 Skill 边界测试 | 与相邻 Skill 职责重叠时归属正确 | 在 When NOT to Use 中正确声明相邻 Skill，重叠场景归属无歧义 |
| 4. 输出质量测试 | 输出结构完整、不编造、标假设 | 输出包含规定章节；无虚构数据/来源；假设与事实已区分；可直接使用 |
| 5. 缺失信息测试 | 输入不足时按规范处理 | 可合理假设则标"假设："继续；会实质影响结果则集中询问一次；不静默编造 |
| 6. 冲突输入测试 | 输入矛盾或自相矛盾时按规范处理 | 指出冲突并合理处理或集中询问，不静默采用错误前提 |
| 7. 事实 / 推断 / 未知区分测试 | 输出严格区分事实、证据支持的推断、未知 / 待验证 | 无无来源数字、不把推断当事实、未知项明确标"待验证" |
| 8. 来源与证据可回溯测试 | 结论可回溯到来源或证据 | 关键结论标注来源 / 证据或标"推断 / 待验证"，来源冲突并列展示 |
| 9. 隐私与敏感信息测试 | 含个人信息的材料不外泄、先匿名化 | 个人信息先匿名化；含个人信息的原始材料不提交外部工具（R05 重点） |
| 10. 文件路径与命名一致性测试 | 源文件 / 目录 / 测试目录命名符合全局规则 | 源文件为 `<skill-name>.md`，目录为 `<编号>-<skill-name>/`，测试目录同名，frontmatter `name` 一致 |

> 统一口径：十类测试为全部 Skill 的默认完整集合。某类测试确实不适用于某个 Skill 时，可在测试记录中标记为 `N/A` 并写明不适用原因，但不得用 `N/A` 逃避第 1 类（触发）、第 2 类（边界）、第 4 类（输出质量）、第 7 类（事实 / 推断 / 未知）或第 10 类（路径与命名一致性）测试。第 10 类适用于全部 Skill，不得省略。涉及用户数据、评论、私信、文件或个人信息的 Skill，第 9 类（隐私与敏感信息）测试不得标记为不适用。测试记录仍须保留预期行为、实际行为、发现的问题、修订内容与复测结果。R01–R05 的测试结果已记录，不因本口径调整而重新执行或改写。

## 3. 测试用例格式

每条测试记录至少包含以下字段：

- 用例编号（如 R01-T1）
- 所属 Skill
- 测试类型
- 输入（用户语料或数据）
- 预期行为
- 实际行为
- 发现的问题
- 修订内容
- 复测结果
- 最终结论（通过 / 不通过）
- 测试日期（YYYY-MM-DD）

## 4. 验收标准

Skill 标记为「已完成」前必须同时满足：

- 十类测试全部通过；确有不适用类型已标 `N/A` 并写明原因，且未跳过触发、边界、输出质量、事实可靠性与路径一致性测试；
- 无未解决的严重触发或边界冲突；
- 不编造数据、来源和事实；
- 输出可直接使用；
- 版本、清单、进度和测试记录已同步。

## 5. 测试记录存放

- 路径：`tests/<编号>-<skill-name>/test-record.md`（`<编号>-<skill-name>` 与 Skill 目录同名）。
- 每个 Skill 一个目录，记录其全部测试用例与结论。
- 测试结果同步登记到 `docs/skill-inventory.md`（测试记录列）与 `progress/PROGRESS.md`。
- 被测试的 Skill 指开发源文件 `skills/<分类>/<编号>-<skill-name>/<skill-name>.md`；若 skill-creator 校验要求 `SKILL.md`，在临时/发布目录生成标准入口文件校验，不改变开发源文件命名（详见 design-spec B10）。

## 6. 定稿与一致性检查（已落地）

Skill 经项目所有者确认定稿后，在生成发布资料与 Agent 兼容运行包前/后，必须完成以下一致性检查（规则见 design-spec B11）：

1. Skill 目录名是否为 `<编号>-<skill-name>`。
2. 开发源文件是否为 `<skill-name>.md`。
3. frontmatter 的 `name` 是否与源文件名、目录中编号后的英文名一致（不含编号）。
4. 版本是否统一（定稿后为 `1.3.0`，后续兼容性优化可升小版本，职责/工作流重大变化才升大版本）。
5. README 能力描述是否与源文件一致，不扩大职责。
6. EXAMPLES 是否符合触发条件与边界，不通过示例扩大职责。
7. 平台发布文案（publish/）是否存在夸大或错误承诺，不把"分析"宣传为"保证结果"。
8. CHANGELOG 是否基于真实版本记录，不虚构未发生的开发行为。
9. 运行包（`dist/standard/<skill-name>/SKILL.md`）正文是否与源文件一致（除兼容性 frontmatter 转换外无内容差异）。
10. 是否存在旧路径引用或旧的人工维护 `SKILL.md` 第二份正文。
11. 是否存在空目录或无用途占位文件。
12. 是否泄露个人路径、账号、密钥或隐私数据。
13. 所有相对链接是否有效。
14. 所有辅助文件是否使用正确版本与名称；skill-creator 校验结果是否真实记录。

- 一致性检查不通过且涉及严重格式错误、内容冲突、安全风险或运行阻塞时，暂停并说明问题，不自行降版本或自行宣布兼容。
- `agents/openai.yaml` 等附加清单文件，仅在 skill-creator 与目标规范真实支持并完成校验时生成；否则不创建，标记为"规范待验证"。

## 7. 与质量复盘手册的衔接（已落地）

- 每个 Skill 的创建前 / 创建后检查以 `templates/skill-review-checklist.md` 为准，配套 `docs/skill-quality-playbook.md`。
- 创建前检查未完成不得进入初稿；创建后检查未通过不得提交审查。
- 后续 Skill 首次交付应直接满足 R01–R05 定稿质量要求，避免重复已确认错误（详见 playbook 第 1 节）。
- 检查清单与复盘手册是质量门禁的一部分，与本节十类测试共同构成验收条件；相关永久创建规则见 `docs/design-spec.md` B12。
